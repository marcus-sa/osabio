User comment: "**Version creation ignores request body — rules can never differ from source**

`handleCreatePolicyVersion` never reads `request.body`. It always creates an exact byte-for-byte copy of the source policy's `title`, `description`, `selector`, `rules`, `human_veto_required`, and `max_ttl`. Because there is also no `PATCH`/`PUT` endpoint for updating a draft policy anywhere in `createPolicyRouteHandlers`, a user who creates a new version through this endpoint has no API surface to modify its rules before activating it.

The typical versioning workflow is: create draft → **edit draft** → activate. The "edit draft" step is missing — callers are stuck with an immutable copy.

Consider parsing an optional request body and merging the provided fields:

```typescript
let overrides: Partial<{ title: string; description: string; rules: PolicyRule[] }> = {};
try {
  const raw = await request.json().catch(() => ({}));
  if (raw && typeof raw === "object") overrides = raw as typeof overrides;
} catch { /* ignore */ }

const { policyId: newPolicyId, version } = await createPolicy(deps.surreal, {
  title: overrides.title ?? sourcePolicy.title,
  description: overrides.description ?? sourcePolicy.description,
  rules: overrides.rules ?? sourcePolicy.rules,
  ...
});
```

Or add a separate `PATCH /policies/:policyId` endpoint, but either way the gap should be closed before this feature is considered complete."

Comment #3:

File: app/src/server/policy/types.ts
Line: 51
User comment: "**TypeScript type contradicts the SurrealDB schema — `description` is `option<string>` in the DB**

The SurrealDB schema defines:
```surql
DEFINE FIELD description ON policy TYPE option<string>;
```

But this PR changes `PolicyRecord.description` from `?: string` to `: string` (non-optional). Pre-existing records in the database that were created before this PR (when `description` was nullable) will have `undefined` here at runtime, even though TypeScript now considers it a guaranteed `string`.

The most concrete impact is in `handleCreatePolicyVersion` (policy-route.ts:388):
```typescript
description: sourcePolicy.description,  // typed string, runtime may be undefined
```
This silently propagates `undefined` as the description of the new draft version, bypassing the `validatePolicyCreateBody` check that would otherwise require a non-empty string.

Either revert to `description?: string` to match the DB schema, or add a `DEFAULT ""` to the SurrealDB field definition and a data migration to backfill existing records.

```suggestion
  description?: string;
```"

Comment #4:

File: app/src/server/orchestrator/worktree-manager.ts
Line: 110
User comment: "**`git worktree prune` and `git worktree add` are not atomic — race condition under concurrent use**

`git worktree prune` modifies the shared `.git/worktrees/` directory for the entire repository. If two `createWorktree` calls run concurrently on the same repo root (which is realistic given that multiple orchestrator sessions can start in parallel):

1. Session A: calls `git worktree prune` — removes any stale metadata
2. Session B: calls `git worktree prune` — also fine so far
3. Session A: calls `git worktree add -b agent/slug-A ...` — creates `.git/worktrees/agent-slug-A/`
4. **Session B**: calls `git worktree prune` again (on the next request) — could prune session A's newly-created but not-yet-fully-initialised worktree entry if it appears stale, causing session A's subsequent git operations to fail

The prune is also unnecessary in the common case where no stale entries exist. Consider making prune a separate, explicitly-invoked maintenance operation, or use a per-repo lock around the `prune` + `add` sequence to make it atomic."
