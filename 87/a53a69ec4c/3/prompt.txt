   - **Push** (current frustration): "What frustrated users enough to request this?"

**Orchestrator Status State Machine:**
```
spawning -> active     (OpenCode session created, first message sent)
active   -> idle       (Agent completed current chat turn, awaiting feedback)
active   -> completed  (Agent finished task, session ended normally)
active   -> error      (OpenCode process error or timeout)
active   -> aborted    (User or system initiated abort)
idle     -> active     (Rejection feedback sent, agent resumes)
idle     -> completed  (User accepts work)
idle     -> aborted    (User or system aborts)
spawning -> error      (OpenCode failed to start within timeout)
```

**Guard Rules:**
- One-agent-per-task enforcement via compound index on `(task_id, orchestrator_status)`
- Task must be in status `["ready", "todo"]` to be assignable
- Task promoted to `in_progress` on successful assignment

**Core Components:**

1. **Assignment Guard** (`assignment-guard.ts`)

   - Enforces one-agent-per-task constraint

   - Returns `AssignmentValidation` or `AssignmentError`

2. **Session Lifecycle** (`session-lifecycle.ts`)
   - Creates OpenCode server via `@opencode-ai/sdk`
   - Manages in-memory `OpenCodeHandle` registry
   - Creates `agent_session` records in SurrealDB
   - Handles process cleanup via `AbortController`
   - Supports status polling and feedback submission

3. **Worktree Manager** (`worktree-manager.ts`)
   - Creates git worktrees under `.brain/worktrees/`
   - Generates diffs between branches
   - Merges and removes branches
   - Validates paths (no arbitrary commands)

4. **Event Bridge** (`event-bridge.ts` - referenced but not fully explored)

   - Transforms OpenCode events to Brain `StreamEvent` types
   - Forwards to Brain SSE registry
   - Implements stall detection

5. **OpenCode Config Builder** (`config-builder.ts`)
   - Builds config object with MCP URLs, auth tokens
   - Sets environment for OpenCode process
   - Injects workspace context

6. **Stall Detector** (`stall-detector.ts`)
   - Monitors inactivity timeout (default: 5 minutes)
   - Counts agent steps (max: 100)
   - Auto-aborts stalled sessions
   - Creates observations on stall detection
   - Emits `agent_stall_warning` events

### 9. Review/Acceptance Flow

**Review Endpoint Response Shape:**

```typescript
type ReviewResponse = {
  agentSessionId: string;
  taskId: string;
  taskTitle: string;
  summary?: string;
  diff: {
    files: Array<{
      path: string;
      status: "added" | "modified" | "deleted";
      additions: number;
      deletions: number;
    }>;
    rawDiff: string;
    stats: {
      filesChanged: number;
      insertions: number;
      deletions: number;
    };
  };
  session: {
    startedAt: string;
    lastEventAt?: string;
    decisionsCount: number;
    questionsCount: number;
    observationsCount: number;
  };
};
```

**Review → Accept Flow:**
1. User calls `GET /api/orchestrator/:ws/sessions/:sessionId/review`
2. Platform generates diff from worktree branch
3. User calls `POST .../sessions/:sessionId/accept`
4. Platform merges branch to main, removes worktree
5. Task status promoted to `done`
6. Session marked as `completed`

**Review → Reject Flow:**
1. User calls `POST .../sessions/:sessionId/reject` with feedback
2. Platform sends feedback as follow-up prompt to OpenCode agent
3. Agent continues in same worktree, same session
4. Cycle returns to monitoring (idle → active)

**Review → Abort Flow:**
1. User calls `POST .../sessions/:sessionId/abort`
2. Platform kills OpenCode process via `AbortController`
3. Worktree removed
4. Session marked as `aborted`
5. Task returns to `ready` status
