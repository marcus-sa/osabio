Workspace-wide task count creates misleading alignment observations

gatherDecisionSignals counts all completed and done tasks across the entire workspace, not just those related to the specific decision being verified. This count is then surfaced in the observation text by compareDecisionConfirmation:

"Decision confirmed: "…". Found 47 completed task(s) in workspace. Alignment check performed."
In practice, those 47 tasks may have nothing to do with the decision being confirmed. A user reading this observation would incorrectly believe that all 47 tasks were checked for alignment with this specific decision, when in reality the deterministic path only counts them, and the LLM path (which does the real per-task check) is conditional on observerModel being set.

The completedTaskCount field should either scope the query to tasks linked to the same project as the decision, or the observation text should not claim an "alignment check was performed" when no per-decision task filtering was done.
