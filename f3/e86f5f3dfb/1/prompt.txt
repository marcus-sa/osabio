Peer review "context" check uses workspace-wide task/decision counts, not observation-linked entities

gatherPeerReviewSignals counts every task and every decision in the workspace and stores them as relatedTaskCount / relatedDecisionCount. compareObservationPeerReview then uses the aggregate:

const hasContext = signals.relatedTaskCount > 0 || signals.relatedDecisionCount > 0;
In any real workspace there will always be at least one task or decision, so hasContext is permanently true. This makes the "inconclusive — no related tasks or decisions found" branch permanently dead and means every observation — even one with no linked entities at all — reaches the confidence-claiming branch.

More importantly, the relatedTaskCount and relatedDecisionCount values that end up in the emitted observation text (e.g. "Found 42 task(s) and 7 decision(s)") imply those entities were cross-checked against this specific observation, which is not the case — they're just workspace totals.

The deterministic peer-review path should use the linkedEntities list already available at the call site (from loadObservationLinkedEntities) rather than a workspace-wide count. If linkedEntities.length === 0, the deterministic path should return inconclusive; if it is non-empty it should reference the actual count.
