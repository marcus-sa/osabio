How Observations Feed Into Suggestions
Observations accumulate in the graph. The suggestion engine (when built) scans observations to generate suggestions:
Pattern aggregation (multiple observations → one suggestion):


3 Observations with category "missing" all targeting the same feature
  → Suggestion: "Feature X has multiple coverage gaps. Consider a review."
  → EVIDENCES edges from Suggestion → each Observation

5 Observations with category "duplication" across 2 projects
  → Suggestion: "Multiple services duplicate [pattern]. Extract shared module."
  → EVIDENCES edges from Suggestion → each Observation
Severity escalation (single high-severity observation → immediate suggestion):


1 Observation with category "contradiction" + severity "high"
  → Suggestion: "Code directly contradicts Decision Y. Resolve before shipping."
  → EVIDENCES edge from Suggestion → the Observation
Cross-project intelligence (observations from different agents across projects):


Coding agent observation in Project A: "Auth service implements custom rate limiting"
Coding agent observation in Project B: "Billing service implements custom rate limiting"  
Background observer: "Both observations target the same pattern"
  → Suggestion: "Extract shared rate limiting middleware" (scope: workspace)
  → EVIDENCES edges → both observations
  → cross_project: [project_a, project_b]
Temporal patterns (observations over time):


Management agent observation week 1: "3 tasks completed, all admin dashboard"
Management agent observation week 2: "4 tasks completed, all admin dashboard"
Management agent observation week 3: "2 tasks completed, all admin dashboard"
  → Suggestion(category: "pivot"): "9 of last 9 completed tasks are admin dashboard. Stated priority is API."
  → EVIDENCES edges → all 3 observations
A single Observation can also be meaningful on its own — it sits on the graph as context, visible in entity detail views and feed, even without a Suggestion.
Relationship Edges
Edge	Direction	When
OBSERVED_IN	Observation → AgentSession	Created during a coding agent session
OBSERVED_IN	Observation → Conversation	Created during a chat agent conversation
OBSERVES	Observation → any entity	Observation targets a specific entity (decision, task, feature, project)
EVIDENCES	Observation → Suggestion	When a Suggestion is generated from accumulated Observations
RELATES_TO	Observation → Observation	When multiple observations are about the same underlying issue (cross-project duplication, recurring pattern)
Pre-Commit Hook Integration
The pre-commit hook (already in the spec) can also generate Observations. When it analyzes the staged diff:
* Detects code contradicting a decision → log_observation(category: "contradiction")
* Detects deprecated dependency added → log_observation(category: "deprecated")
* Detects missing test coverage for changed code → log_observation(category: "missing")
These are automatic Observations — the agent didn't explicitly call the tool, the hook generated them from diff analysis.
Session Summary Integration
AgentSession entity should include observations_logged: string[] (Observation entity IDs) alongside the existing decisions_made, questions_asked, and tasks_progressed arrays. The SessionEnd summary counts observations: "Logged 3 observations (1 contradiction, 2 missing)."
