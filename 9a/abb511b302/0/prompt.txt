Here is the proposal for the **`trace`** table and the refactor.

---

### 1. The Decision: Create a new `trace` table

I recommend **Option 1 (New Table)**.

* **Why:** A "Trace" is a specific technical artifact that represents a "Unit of Thought." While a `message` is for humans and a `session` is for duration, a **Trace** is the forensic record of the logic.

### 2. The Refactored Schema (SurrealQL)

```sql
-- 1. The Trace Table: The "Black Box Recorder"
DEFINE TABLE trace SCHEMAFULL;
DEFINE FIELD type ON trace TYPE string ASSERT $value IN ['tool_call', 'message', 'subagent_spawn', 'logic_step'];
DEFINE FIELD actor ON trace TYPE record<identity>;
DEFINE FIELD content ON trace TYPE object; -- The actual input/output or log
DEFINE FIELD parent_trace ON trace TYPE option<record<trace>>; -- For nested hierarchies

-- 2. The Intent Table (The Refactor)
DEFINE TABLE intent SCHEMAFULL;
-- We use TYPE record<trace> to ensure referential integrity
DEFINE FIELD trace_id ON intent TYPE record<trace>; 

```

This schema allows your **Authorizer Agent** to walk the tree to see exactly why an intent was generated:

* **Level 1:** `trace:root` (The Subagent spawn).
* **Level 2:** `trace:msg_1` (Parent: `trace:root`).
* **Level 3:** `trace:tool_1` (Parent: `trace:msg_1`).

If the agents talk directly to an LLM, the "Brain" is just an observer. But if the Brain acts as a Proxy, it becomes the Enforcement Officer. This is the only way to truly solve the "Lying Agent" problem and the "Context Drift" problem at scale.

Here is why the LLM Proxy is the most important architectural decision for your Business OS.

1. The "Automatic Enforcement" Layer

When the Brain acts as a proxy, it can run Pre-Request Interceptors. * The Flow: Agent sends a request → Brain Interceptor checks the identity and active policy nodes → Brain injects the "Learning" nodes and "Rules" into the system prompt automatically.

Why this wins: You don't have to trust the agent's code to "remember" the rules. The rules are injected at the wire level. The agent physically cannot see the LLM without the guardrails being attached.

2. The "Automatic Capture" (Self-Logging Graph)

If the Brain sits in the middle, it can run Post-Response Parsers. * Every time the LLM responds, the Brain can scan the text for Intents, Tasks, or Observations.

The Result: The Knowledge Graph updates itself passively. You don't need the agent to explicitly call save_to_graph(). The proxy sees the "Intent" in the LLM's thought process and creates the node in SurrealDB automatically.

3. The "Stateful" Proxy: Beyond Stateless Chat

Standard LLM APIs are stateless. By using a proxy, you give the LLM "Virtual Memory":

Context Injection: The proxy can look at the agent's identity and automatically pull the last 5 relevant Observations from the graph and "stuff" them into the prompt.

Cost/Token Control: The proxy can enforce Budget Policies. If an agent is hallucinating in a loop and burning tokens, the Brain kills the connection once it hits a "Policy Ceiling."

In 2026, this is often implemented as a sidecar container or a custom MCP (Model Context Protocol) Server.

If you don't build a proxy, your agents will eventually "drift." They will find ways to bypass your policies because their code is what controls the LLM call.

By building the proxy, you turn the LLM into a "Utility" and the Brain into the "Controller." This is how you build a system that can be formally verified. This is the classic trade-off: Reliability vs. Velocity. In a 2026-era Business OS, you shouldn't pay the "Tax of Certainty" on every single token. If an agent is just refactoring a CSS file or writing a unit test, a full Graph-governed proxy check is overkill. Here is how you solve the Latency/Cost problem without sacrificing the "Brain" architecture.

1. The "Dynamic Interception" Strategy

You don't proxy everything. You use Tiered Routing based on the agent's current Authority Scope.

Fast Path (Direct): For "Low-Stakes" tasks (Coding, Drafting, Research). The agent talks directly to a cheaper, faster model (like Gemini 3.5 Flash or Claude 3 Haiku). No proxy, minimal logging.

Secure Path (Proxied): For "High-Stakes" tasks (Financials, PII, Production Deployments). The proxy intercepts the call, injects the Policy Node, and runs the Intent Verification.

2. The "Sidecar" vs. "Gateway" Model

To kill the latency, you move the "Brain" logic closer to the agent.

The Solve: Instead of a central cloud gateway, you run a Local Sidecar (a tiny, high-performance Go or Rust binary) inside the agent's sandbox.

It handles the Policy Check in microseconds by keeping a cached copy of the relevant SurrealDB nodes. It only pings the "Main Brain" if it sees a potential violation.

3. "Coding" as the Exception

You mentioned coding—this is the perfect use case for Post-Hoc Grounding.

The Logic: Let the agent code at full speed. Don't intercept the LLM calls.

The Catch: The agent cannot Merge or Execute the code until it passes the Observer Agent's audit.

The Result: The developer/agent feels zero latency during the "Creative" phase, but the "Execution" phase is still 100% governed.

4. Cost Optimization: The "Cerebral Cache"

If the Brain acts as a proxy, it can implement Semantic Caching.

If Agent A asks a coding question that Agent B already asked 10 minutes ago, the Brain returns the cached response from SurrealDB without ever hitting the LLM API.

The Result: The proxy actually saves you money and reduces latency for repetitive tasks.

If you try to make a "Perfectly Safe" system, you'll build something so slow that no one will use it. If you build a "Perfectly Fast" system, you'll build the LinkedIn "Lying Agent."

The Play: Use the proxy as a Variable Valve.

10% Latency/Cost Increase for 100% Truth on Financials.

0% Latency/Cost Increase for "Vibes-based" coding and research.
