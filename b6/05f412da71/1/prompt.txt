User comment: "**Rule `id` field not validated server-side**

`validateRule` checks `effect` and `condition` but never validates the presence of `id`. The `PolicyRule` type in `types.ts` declares `id: string` as required, and `VersionDiffView.diffRules` builds lookup maps keyed on `r.id`:

```typescript
const oldById = new Map(oldRules.map((r) => [r.id, r]));
```

Any rule stored without an `id` (possible via direct API calls — the UI's `RuleBuilder` always generates one, but nothing enforces this server-side) would map to key `undefined`. Multiple id-less rules would collapse onto the same slot, silently producing incorrect diff output.

Add an `id` check to `validateRule`:

```suggestion
function validateRule(rule: unknown, index: number): string[] {
  if (!isPlainObject(rule)) {
    return [`rule[${index}]: must be an object`];
  }

  const errors: string[] = [];

  // ID
  if (typeof rule.id !== "string" || (rule.id as string).trim() === "") {
    errors.push(`rule[${index}]: id must be a non-empty string`);
  }

  // Effect
  if (!VALID_EFFECTS.has(rule.effect as string)) {
    errors.push(`rule[${index}]: effect must be "allow" or "deny"`);
  }

  // Condition
  errors.push(...validateCondition(rule.condition, index));

  return errors;
}
```"
