User comment: "**`supports` edge created before LLM evaluation completes**

`createSupportsEdge` persists an immutable graph edge linking `intentId → objectiveId` **before** the LLM evaluation runs (and before its outcome is known). The edge comment says "Immutable once created."

If the LLM evaluation subsequently times out or throws, the function still returns `APPROVE` with `policy_only: true` — meaning the intent proceeds, so the edge is not technically dangling in normal flow. However, if the *caller* has not yet committed the intent record to the database when it calls `evaluateIntent` (i.e., the intent record doesn't exist yet), the supports edge will reference a non-existent `intent` record.

More critically, the alignment step runs before the timeout `AbortController` is initialised (line 140). This means `findAlignedObjectives` is itself not subject to the `timeoutMs` budget, so a slow KNN query could push the total wall-clock time beyond `timeoutMs` with no cancellation path, and the `supports` edge write is similarly unguarded.

Consider either:
1. Moving the alignment step inside the LLM `try` block so it shares the `AbortController`, or
2. Passing the `AbortSignal` to `findAlignedObjectives` and `createSupportsEdge`."

User comment: "**Unbounded `listBehaviors` call with in-memory workspace filter**

When `identity_id` is provided, `listBehaviors` fetches **all** behavior records for the identity across **all** workspaces and then filters in memory on line 63. `listBehaviors` has no `LIMIT` clause (unlike `listWorkspaceBehaviors` which defaults to `limit = 50`). For an identity active in multiple workspaces with many sessions, this will load an unbounded number of records before discarding the cross-workspace ones.

Either push the workspace filter into the query, or add a limit parameter to `listBehaviors`, for example:

```
SELECT * FROM behavior
  WHERE id IN $ids AND workspace = $ws
  ORDER BY created_at DESC LIMIT $limit;
```

This is especially important given the append-only record model — the table is designed to grow without bound."

User comment: "**`listObjectives` missing LIMIT clause — unbounded query**

Both query variants in `listObjectives` have no `LIMIT` clause, fetching every objective in the workspace in a single round-trip. For workspaces with many objectives this will grow without bound, consistent with the same pattern already flagged for `listBehaviors`.

Consider adding a `limit` parameter (defaulting to something sensible, e.g. `100`) and applying it to both queries:

```suggestion
export async function listObjectives(
  surreal: Surreal,
  workspaceId: string,
  status?: ObjectiveStatus,
  limit = 100,
): Promise<ObjectiveRecord[]> {
  const workspaceRecord = new RecordId("workspace", workspaceId);

  const query = status
    ? `SELECT * FROM objective WHERE workspace = $ws AND status = $status ORDER BY created_at DESC LIMIT $limit;`
    : `SELECT * FROM objective WHERE workspace = $ws ORDER BY created_at DESC LIMIT $limit;`;
  const rows = (await surreal.query(query, { ws: workspaceRecord, status, limit })) as Array<ObjectiveRecord[]>;
  return rows[0] ?? [];
}
```"
