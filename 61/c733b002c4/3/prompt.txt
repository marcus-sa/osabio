Fragile 40-character prefix deduplication causes false positives

isAlreadyObserved compares only the first 40 characters of each observation text to determine duplicates. This is prone to false positives: two observations that start with the same prefix but describe entirely different issues will be silently suppressed.

For example, all stale-blocked task observations begin with "task blocked for" followed by a variable number of days, meaning an observation for a 14-day blocked task and a 21-day blocked task on different entities could be considered duplicates if their titles happen to share 40 characters.

A safer approach is to include the entity ID in the deduplication key, or use a more specific check such as a minimum similarity threshold or entity-aware hashing.

Sequential LLM calls per completed task — N+1 latency problem

verifyDecision iterates over all completed tasks (up to 20 from queryCompletedTasks) and issues a separate, sequential generateVerificationVerdict LLM call for each one with no concurrency:

for (const task of completedTasks) {
  ...
  const llmVerdict = await generateVerificationVerdict(observerModel, context, deterministicResult);
  ...
}
With 20 tasks and a 30-second timeout per call, this path can block for up to 600 seconds. Even under normal latency (~2–5 s per call) a workspace with 20 completed tasks will stall the route handler for 40–100 seconds. Promise.allSettled should be used to run the LLM calls concurrently, or the tasks should be batched.
