## New Dependencies

| Technology | Version | License | Role | Justification |
|-----------|---------|---------|------|---------------|
| @opencode-ai/sdk | latest | MIT | OpenCode server spawn + client API | Required to interact with OpenCode coding agent. No alternative SDK exists for OpenCode. Provides `createOpencodeServer()` (child process spawn) and `createOpencodeClient()` (session/event management). |
| @opencode-ai/plugin | latest | MIT | Plugin type definitions + `tool()` helper | Required to create typed OpenCode plugins with custom tools. Provides `Plugin` type, `tool()` factory, and `tool.schema` for argument definitions. Dev dependency for the plugin source. |

### @opencode-ai/sdk API Surface Used

**Server** (`@opencode-ai/sdk/server`):
- `createOpencodeServer(options)` -- Spawns `opencode serve` as child process via `node:child_process.spawn()`
- Options: `hostname`, `port`, `signal` (AbortSignal), `timeout`, `config`
- Returns: `{url, close()}`

**Client** (`@opencode-ai/sdk/client`):
- `createOpencodeClient(config)` -- Creates HTTP client for OpenCode API
- `client.session.create({directory})` -- Creates coding session in worktree
- `client.session.chat({sessionID, message})` -- Sends task prompt / feedback
- `client.session.abort(sessionID)` -- Aborts active session
- `client.session.status(sessionID)` -- Polls session state
- `client.event.subscribe(sessionID)` -- SSE event stream per session
- `client.global.event()` -- Global SSE event stream

### @opencode-ai/plugin API Surface Used

**Plugin type** (`@opencode-ai/plugin`):
- `Plugin` type -- async function receiving context, returning hooks + tools
- Context: `{ project, client, $, directory, worktree }` -- project info, SDK client, Bun shell, paths
- Hook events: `session.created`, `session.idle`, `session.updated`, `message.updated`, `tool.execute.before`, `experimental.session.compacting`

**Tool helper** (`@opencode-ai/plugin`):
- `tool({ description, args, execute })` -- creates a custom tool definition
- `tool.schema.string()`, `.number()`, `.boolean()`, `.enum()`, `.array()`, `.object()` -- Zod-like schema builders for tool arguments
- `execute(args, context)` -- receives parsed args + `{ directory, worktree }` context, returns string result

### CLI HTTP Client Reuse

The Brain OpenCode plugin imports the existing `BrainHttpClient` from `cli/http-client.ts`. This class:
- Manages OAuth token storage and automatic refresh (60s before expiry)
- Constructs URLs with workspace ID prefix (`/api/mcp/:workspaceId/`)
- Handles errors with descriptive messages
- Works identically for both Claude Code MCP server and OpenCode plugin

The plugin is a thin adapter: each custom tool maps to one `httpClient.post()` call using the same endpoint paths as the MCP server.

## No Additional Dependencies Required

| Concern | Solution | Why No New Dep |
|---------|----------|----------------|
| Git worktrees | `Bun.spawn()` / `node:child_process.exec()` | Git CLI already available on host. 5 commands total (`worktree add`, `worktree remove`, `diff`, `merge`, `branch -D`). No wrapper library justified. |
| Process management | `AbortController` + `createOpencodeServer({signal})` | Native Node.js API. OpenCode SDK accepts AbortSignal directly. |
| SSE event bridging | Existing `SseRegistry` | Transform OpenCode events to `StreamEvent` union. No new SSE library needed. |
| Diff parsing | Raw `git diff` output | Walking skeleton: return raw diff text. Future: parse unified diff format with simple string splitting -- no library needed for <100 line parser. |
| Config injection | `OPENCODE_CONFIG_CONTENT` env var | OpenCode SDK reads config from this env var. JSON serialization only. |

## OpenCode Config Structure

### User-facing config (generated by `brain init --opencode`)

The `opencode.json` in the project root references the Brain plugin. The plugin is loaded from the local `.opencode/plugins/` directory:

```
// opencode.json
{
  "$schema": "https://opencode.ai/config.json",
  "instructions": ["./OPENCODE.md"]
}
```

```
// .opencode/plugins/brain.ts -- loaded automatically from plugin directory
// .opencode/package.json -- declares @opencode-ai/plugin dependency
```

### Orchestrator-injected config (for spawned agents)

When the orchestrator spawns an OpenCode server for task assignment, it passes config via `OPENCODE_CONFIG_CONTENT`:

```
{
  // Plugin loaded from worktree's .opencode/plugins/brain.ts
  // Auth injected via environment variables:
  //   BRAIN_SERVER_URL, BRAIN_WORKSPACE_ID, BRAIN_ACCESS_TOKEN
  "provider": {
    // Inherited from OpenCode's own config or set explicitly
  },
  "model": {
    // Agent model configuration
  },
  "permission": {
    // Auto-approve all tools for unattended agent operation
    "bash": "allow",
    "edit": "allow",
    "write": "allow"
  }
}
```

The Brain plugin reads `BRAIN_SERVER_URL`, `BRAIN_WORKSPACE_ID`, and `BRAIN_ACCESS_TOKEN` from the environment when initialized by the orchestrator. For interactive use, it reads from `~/.brain/config.json` (same as Claude Code integration).

## Build and Runtime Impact

- **Bundle**: `@opencode-ai/sdk` is a lightweight HTTP client + child process spawner. Minimal bundle impact.
- **Runtime**: Each active agent adds one child process. Memory ~100-200MB per OpenCode process.
- **Startup**: No impact. OpenCode processes spawned lazily on first assignment.
- **Tests**: OpenCode SDK can be mocked at the function boundary (factory functions return plain objects). No test infrastructure changes.
