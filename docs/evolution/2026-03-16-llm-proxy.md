# Evolution: LLM Proxy -- Transparent Passthrough with Intelligence Layer

**Date:** 2026-03-16
**Branch:** `marcus-sa/llm-proxy-research`
**Commits:** 19 (12 feature + 3 docs/design + 2 research + 1 refactor + 1 fix)

## Summary

Added a transparent LLM proxy that intercepts Anthropic API calls from coding agents (Claude Code, Cursor, etc.), captures full telemetry as graph-native trace nodes, enforces workspace policies (model access, budget, rate limits), injects knowledge graph context into system prompts, and extends the Observer with per-trace contradiction detection and session-end pattern synthesis. The proxy sits in-process (no sidecar), forwards requests transparently, and performs all intelligence work asynchronously so it never blocks response delivery.

Two architectural decisions anchor the design:
- **ADR-050**: UUIDv5 content-derived conversation hashing for deterministic correlation across all clients without client cooperation
- **ADR-051**: Observer-owned detection -- the proxy captures data but performs zero analysis; all intelligence lives in the Observer system triggered by SurrealDB EVENTs

## Architecture Overview

```
Coding Agent (Claude Code / Cursor / etc.)
  |
  v
Brain Proxy (in-process Bun route)
  |- Identity resolution (metadata.user_id + headers)
  |- Session ID resolution (session_{uuid} pattern)
  |- Conversation hash (UUIDv5 content-derived)
  |- Policy evaluation (model access, budget, rate limit)
  |- Context injection (<brain-context> XML in system prompt)
  |- Request forwarding (transparent SSE passthrough)
  |- Async trace capture (llm_call graph nodes + cost computation)
  v
Anthropic API
  |
  v (SurrealDB EVENT on trace creation)
Observer
  |- Per-trace analysis (contradiction + missing decision detection)
  |- Session-end synthesis (approach drift, accumulated contradictions)
  |- Reverse coherence scan (implementations without decisions)
```

Key design principles applied:
- Pure functions for identity resolution, session ID parsing, conversation hashing, cost calculation, rate limiting, context injection, policy evaluation
- Fail-open on all intelligence features -- proxy always forwards the request
- Async trace capture via `deps.inflight.track()` -- never blocks response delivery
- Two-step KNN pattern for Observer vector searches (SurrealDB v3.0 HNSW + B-tree workaround)
- In-memory caching with TTL for rate limits, spend aggregation, context candidate pools, and intelligence config

## Key Architectural Decisions

| ADR | Title | Summary |
|-----|-------|---------|
| ADR-040 | In-process module | Proxy runs as a Bun route handler, not a sidecar or separate process |
| ADR-041 | Async trace capture | Trace writes use inflight tracker, never block response delivery |
| ADR-042 | LLM trace as graph entity | Traces are first-class graph nodes with RELATE edges to sessions, workspaces, tasks, policies |
| ADR-043 | Policy engine reuse | Reuses existing `evaluatePolicyGate()` pipeline with llm_call context |
| ADR-044 | Local pricing table | Static TypeScript config for cost calculation, no external API dependency |
| ADR-045 | Multi-provider via separate routes | Future providers get their own route handlers, not a unified abstraction |
| ADR-046 | Context injection append-only | Brain context appended to system prompt, never replaces existing content |
| ADR-047 | Contradiction detection hybrid | Proxy captures, Observer analyzes (later refined by ADR-051) |
| ADR-048 | Observer session-end analysis | Cross-trace pattern synthesis triggered by session end EVENT |
| ADR-049 | Session lifecycle inactivity timeout | Session boundaries detected via inactivity threshold |
| ADR-050 | UUIDv5 conversation hash | Content-derived deterministic conversation correlation across all clients |
| ADR-051 | Observer-owned detection | Proxy performs zero analysis; all intelligence in Observer via SurrealDB EVENTs |

## Lessons Learned

1. **Observer-owned detection is the right boundary.** Keeping the proxy as a pure data capture layer and pushing all analysis to the Observer (ADR-051) simplified the proxy pipeline, made intelligence features independently testable, and avoided coupling proxy latency to LLM verification calls. The SurrealDB EVENT mechanism (ASYNC, RETRY 3) provides reliable decoupled triggering.

2. **UUIDv5 for content-derived correlation is robust.** Using deterministic hashing (system_prompt + first_user_message) for conversation IDs eliminates the need for client cooperation or session state. The same conversation produces the same ID regardless of which coding agent sends it, and UUIDv5 will not collide with UUIDv4 used by the web UI.

3. **Fail-open is essential for proxy intelligence.** Every intelligence feature (context injection, conversation hash, policy evaluation) is wrapped in fail-open error handling. The proxy always forwards the request. This was validated by the review process which specifically tested error paths.

4. **In-memory caching with TTL covers the proxy use case.** Rate limiting, spend aggregation, context candidate pools, and intelligence config all use simple in-memory Maps with TTL expiry. No external cache (Redis) needed for single-process deployment. The rate limiter pruning finding from review was a good catch -- without it, the Map would grow unbounded for high-traffic workspaces.

5. **Pre-implementation roadmap review pays for itself.** The crafter review caught 11 issues (6 blocking, 5 high) in the roadmap before any code was written. Fixing acceptance criteria and specifying exact queries upfront prevented rework during implementation. The two-iteration review process added ~30 min but saved significantly more.
