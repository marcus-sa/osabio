---

## Summary

Added a governed domain expertise layer between Tools (functional capabilities) and Learnings (reactive corrections). Skills are metadata-and-source-pointer records in the knowledge graph that control which domain expertise is available to sandbox agents. The implementation spans full-stack: SurrealDB schema, backend CRUD/lifecycle API, agent creation extension, 3-step wizard UI, and Skill Library UI.

## Business Context

Workspace admins needed a way to equip agents with domain expertise (security auditing, code review, compliance checks) without manually configuring tool grants. Skills bridge the gap: assign a skill to an agent, and the agent automatically gains the tools that skill requires. The North Star KPI is >50% of sandbox agents with skills assigned.

## What Was Delivered

### Backend
- **Migration 0084**: `skill` table + 5 relation tables (`skill_requires`, `possesses`, `skill_supersedes`, `skill_evidence`, `governs_skill`)
- **Skill CRUD API**: Create, list, detail, update, delete with source-reference architecture
- **Skill Lifecycle**: `draft → active → deprecated` state machine via explicit endpoints
- **Agent Creation Extension**: Atomic transaction includes `possesses` + `can_use` edges
- **Policy Governance**: `governs_skill` relation traversal in skill detail

### Frontend
- **3-Step Agent Wizard**: Config → Skills checklist → Tools review → Create
- **Skill Library**: List page with status filter, detail page, create form
- **Skill Hooks**: `useSkills`, `useSkillDetail`, `useCreateSkill`, `useActivateSkill`, `useDeprecateSkill`

### Test Coverage
- 31 acceptance test scenarios across 5 test files
- 3 walking skeletons + 28 focused scenarios (17 happy, 8 error, 6 edge)
- 45% error+edge ratio (exceeds 40% threshold)

## Key Decisions

### From DISCOVER Wave
- **D1**: 3-step wizard (Config > Skills > Tools) mirrors conceptual layers
- **D2**: Minimal checklist for skill assignment — right for <20 skills
- **D3**: Two-section tool display (skill-derived read-only + manual selection)
- **D4**: Steps 2 and 3 are skippable
- **D5**: Runtime selection consolidated into Step 1

### From DISCUSS Wave
- **D6**: 10 user stories: Walking Skeleton (7) + Release 1 (3)
- **D8**: Atomic creation separated from wizard UI
- **D9**: Session lifecycle in Walking Skeleton (riskiest assumption first)
- **D10**: Policy governance split: relation in WS, evaluation in R1
- **D13**: North Star KPI: >50% sandbox agents with skills

### From DESIGN Wave
- **D14**: Extend `createAgentTransaction` atomically (ADR-084)
- **D15**: Source-reference architecture — metadata + pointer, never file content (ADR-085)
- **D16**: Skill module follows learning system pattern (ADR-086)
- **D17**: Non-UNIQUE index on workspace+name (SurrealDB v3.0.4 bug workaround)
- **D20**: Name uniqueness at application layer (pre-validation query)
- **D21**: Lifecycle via explicit endpoints, not generic status PUT
- **D22**: Skill-derived tools resolved client-side

### From DISTILL Wave
- **D24**: 31 scenarios organized as 3 walking skeletons + 28 focused
- **D27**: Tests drive through HTTP API exclusively (hexagonal boundary)
- **D28**: WS-1 enabled first to prove end-to-end

## Lessons Learned

1. **Identity linking required in tests**: All acceptance tests initially returned 500 errors because `identity_person` edges weren't established. Created `linkUserToWorkspaceIdentity` helper — pattern must be applied to all test files touching session-based auth.
2. **SurrealDB UNIQUE index bug**: Non-UNIQUE index + app-layer validation is the reliable workaround for v3.0.4.
3. **Vertical slice > sequential phases**: Proving end-to-end integration early (walking skeleton) reduced risk more than completing all backend before starting frontend.
4. **Frontend test infrastructure gap**: UI component tests skipped due to no unit test infrastructure for MVP. Documented as explicit trade-off, not silent omission.
